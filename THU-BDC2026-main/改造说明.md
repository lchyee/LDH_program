# THU-BDC2026 多维数据改造说明

> **日期**: 2026-04-25  
> **改造人**: Lu Sanqian (DGH分支)  
> **目的**: 在原有纯量价数据基础上，引入4大新数据维度，提升股票排序预测模型的性能上限

---

## 一、为什么需要新数据维度

### 1.1 原有数据的天花板

改造前，模型仅使用 baostock 日线 K 线的 12 个基础字段：

```
股票代码, 日期, 开盘, 收盘, 最高, 最低, 成交量, 成交额, 振幅, 涨跌额, 换手率, 涨跌幅
```

通过 utils.py 中的特征工程，扩展为 197 维特征（158个Alpha因子 + 39个技术指标）。

**核心问题**：这 197 维特征全部派生自量价数据。量价数据本质上是"市场交易结果的录像带"——它记录了价格和成交量如何变化，但无法回答以下问题：

| 缺失信息 | 导致的问题 |
|----------|-----------|
| 公司值多少钱？ | 不知道 PE 是 5 还是 500，无法判断估值高低 |
| 公司赚钱效率如何？ | 不知道 ROE 是 2% 还是 20%，无法区分贵但优秀的公司与便宜但平庸的公司 |
| 钱是谁在买卖？ | 不知道今天是机构在买入还是游资在拉升 |
| 股票属于什么行业？ | 银行股和科技股放在一起排序，缺乏行业对照组 |
| 大盘环境如何？ | 降息周期还是加息周期？市场恐慌还是亢奋？ |
| 市场情绪如何？ | 今天是百股涨停的牛市还是千股跌停的熊市？ |

### 1.2 改造目标

引入 **4 大新数据维度**，设计新增 **21 个特征字段**，但受限于数据源实际可用性，当前**实际可用 10 个新字段**（详见第十节排查记录），将模型输入从 197 维扩展到约 **206 维**（含行业 Embedding 则为 207 维）：

| 维度 | 设计字段数 | 当前可用 | 来源 | 备注 |
|------|-----------|---------|------|------|
| 日频估值 (PE/PB/PS/PCF) | 4 | **4** | baostock K线扩展字段 | ✅ 零额外API开销 |
| 行业分类 (Embedding) | 1 | **1** | baostock query_stock_industry | ✅ 不经过StandardScaler |
| 季频基本面 (ROE/成长/偿债) | 6 | **0** | baostock 财务报表API | ❌ stock_fundamental.py 未成功运行 |
| 宏观指标 (CPI/PMI/M2) | 4 | **0** | akshare 国家统计局/央行 | ❌ API远程抓取失败 |
| 市场情绪 (QVIX/涨停) | 5 | **5** | akshare | ✅ QVIX/SHIBOR/涨停统计可用 |

## 二、整体数据流

### 2.1 新数据管道

```
┌──────────────────────────────────────────────────────┐
│                      数据抓取层                       │
├──────────────┬───────────────┬──────────────────────┤
│ get_stock_   │ stock_        │ stock_       │ market_│
│ data.py      │ industry.py   │ fundamental.py│ data.py│
│ (量价+PE/PB) │ (行业分类)     │ (ROE/成长)   │(宏观/情绪)│
├──────────────┴───────────────┴──────────────┴───────┤
│                                                       │
│                   merge_all_data.py                    │
│                 合并所有维度到一个CSV                    │
│                                                       │
├───────────────────────────────────────────────────────┤
│                   split_train_test.py                  │
│                按日期切分 train.csv / test.csv          │
│                                                       │
├───────────────────────────────────────────────────────┤
│                     train.py / predict.py              │
│                    (特征工程 + 模型训练/推理)           │
└───────────────────────────────────────────────────────┘
```

### 2.2 关键设计决策

**1. 数据存储策略：先分后合**
- 每个维度单独抓取、单独存储为独立 CSV
- 最后通过 `merge_all_data.py` 按 (股票代码, 日期) 合并为一张大表
- **原因**：各数据源稳定性不同（baostock稳定，akshare有时候抽风），分开抓取可以独立重试失败的部分

**2. 行业使用 Embedding 而非 One-Hot**
- 行业名称 → 整数索引 → `nn.Embedding(28, 16)` 映射为 16 维向量
- 拼接到 Transformer 编码后的时序特征上
- **原因**：Embedding 可以让模型学到行业之间的相似性（例如"银行"和"非银金融"应该比"银行"和"医药"更接近）

**3. 季频数据前向填充到日频**
- ROE、净利增长率等财务数据每季度才公布一次
- 在 `merge_all_data.py` 中按"最近一个已公布季度"填充到每个交易日
- **原因**：确保不会引入未来信息（不能用还未公布的财报去预测过去）

**4. 兼容降级策略**
- train.py 和 predict.py 会先检查 `stock_data_merged.csv` 是否存在
- 如果不存在，自动回退到 `train.csv`（原有12字段的基础数据）
- **原因**：新数据源需要额外抓取，不能强依赖；保证旧流程随时可用

## 三、各数据维度详解

### 3.1 日频估值 — PE / PB / PS / PCF

**修改文件**: `get_stock_data.py`

**实现方式**：在 baostock 的 `query_history_k_data_plus()` 调用中，fields 参数追加 4 个字段：

```python
# 原来
"date,code,open,high,low,close,preclose,volume,amount,turn,pctChg"

# 改为
"date,code,open,high,low,close,preclose,volume,amount,turn,pctChg,
 peTTM,pbMRQ,psTTM,pcfNcfTTM"
```

| 字段 | 中文名 | 含义 | 越X越___ |
|------|--------|------|----------|
| peTTM | 市盈率 TTM | 股价 / 近12个月每股收益 | 高=贵，低=便宜 |
| pbMRQ | 市净率 MRQ | 股价 / 最新季报每股净资产 | <1=破净，银行股常见 |
| psTTM | 市销率 TTM | 股价 / 近12个月每股营收 | 营收型企业估值指标 |
| pcfNcfTTM | 市现率 TTM | 股价 / 近12个月每股自由现金流 | 现金流型企业估值指标 |

**零额外API开销**：这4个字段是 baostock K线接口内置支持的，不增加网络请求次数。

---

### 3.2 行业分类

**新建文件**: `data/stock_industry.py`

**数据源**: baostock `query_stock_industry()` — 申万行业分类标准

**输出**: `data/stock_industry.csv`

```
股票代码, 行业名称, 行业分类标准
600000, 银行, 申万一级行业
000001, 银行, 申万一级行业
600519, 食品饮料, 申万一级行业
```

申万一级行业约 28 个类目，如：银行、非银金融、医药生物、电子、计算机、食品饮料、汽车等。

**为何重要**：
- A股有显著的**板块轮动效应**—同行业股票往往同涨同跌
- 模型可以用行业信息做两件事：
  1. 学习行业内相对排序（同行业内谁的涨幅更高）
  2. 跨行业比较（当前是银行强还是科技强）

**模型中的处理**：
- 行业名称 → 整数索引（train.py 构建 vocab）
- 进入 `nn.Embedding(num_industries, 16)` → 16维向量
- 拼接到时序特征后，一起进入 CrossStockAttention

### 3.3 季频基本面 — ROE / 成长性 / 偿债能力

**新建文件**: `data/stock_fundamental.py`

**数据源**: baostock 财务报表 API

| API | 提取字段 | 字段含义 |
|-----|----------|----------|
| `query_profit_data()` | ROE_avg, 净利率, EPS_TTM | 盈利能力 |
| `query_growth_data()` | 净利同比, 权益同比 | 成长能力 |
| `query_balance_data()` | 资产负债率 | 财务杠杆风险 |

**为什么 ROE 是核心指标**：
- ROE（净资产收益率）= 净利润 / 净资产
- 巴菲特的核心选股标准之一：长期 ROE > 15% 说明公司有"护城河"
- 高 ROE + 低 PE = 可能的低估机会

**为什么需要成长性**：
- 净利同比增长 > 30% 的高成长公司往往享受更高估值
- 净利连续下滑的公司即使 PE 很低也可能是"价值陷阱"

**为什么需要资产负债率**：
- 负债率 > 80% 的公司（如地产、建筑）在经济下行时风险极高
- 银行股天然高杠杆（负债率 > 90%）—需要结合 ROE 判断

**前向填充逻辑**（merge_all_data.py）：
```
Q1财报(3/31) → 覆盖 4/1 ~ 6/29 的所有交易日
Q2财报(6/30) → 覆盖 7/1 ~ 9/29 的所有交易日
...
```

### 3.4 宏观指标 + 市场情绪

**新建文件**: `data/market_data.py`

**数据源**: akshare 国家统计局、央行、中证指数等公开数据

#### 宏观指标

| 字段 | 来源 | 频率 | 含义 |
|------|------|------|------|
| CPI_同比 | 国家统计局 | 月频 | 通胀水平，影响货币政策 |
| CPI_环比 | 同上 | 月频 | 短期物价变化 |
| PMI_制造业 | 统计局/财新 | 月频 | >50=扩张，<50=收缩，经济的"体温计" |
| M2_同比 | 央行 | 月频 | 货币供应增速，流动性宽松→利好股市 |
| SHIBOR_ON | 银行间市场 | 日频 | 隔夜拆借利率，市场流动性松紧的实时信号 |

**为什么宏观指标对股票排序有用**：
- M2增速上行 → 市场上钱多 → 利好高贝塔（弹性大）的成长股
- 利率下行 → 资金成本降低 → 利好高负债行业（地产、基建）
- PMI < 50 → 经济收缩 → 资金偏好防御性行业（医药、公用事业）

#### 市场情绪

| 字段 | 来源 | 含义 |
|------|------|------|
| QVIX | 50ETF期权隐含波动率 | 中国版"恐慌指数"，越高=市场越恐惧 |
| 涨停家数 | 每日涨停股池统计 | 市场热度指标，>100家=市场亢奋，<20家=低迷 |
| 连板均值 | 同上 | 涨停股的连续涨停天数均值，反映短线情绪 |
| 封单资金均值 | 同上 | 涨停板上的封单量，反映资金抢筹意愿 |

**为什么重要**：
- 极端情绪日（QVIX 飙升 + 百股跌停）→ 应该空仓而非选股
- 情绪回暖（QVIX 回落 + 涨停家数增加）→ 可能是加仓时机
- 这些是不可从量价数据推断的"环境变量"

## 四、模型改动说明

### 4.1 StockTransformer 新增行业 Embedding

**修改文件**: `code/src/model.py`

```python
# 新增层
self.industry_embedding = nn.Embedding(num_industries, industry_emb_dim, padding_idx=0)
self.industry_proj = nn.Linear(d_model + industry_emb_dim, d_model)

# forward() 中：
# 1. 时序编码 → 特征注意力聚合 → [batch*N, d_model]
# 2. 行业 Embedding → [batch*N, industry_emb_dim]
# 3. 拼接 → Linear → [batch*N, d_model]
# 4. CrossStockAttention → ranking head
```

**设计考量**：
- 行业 Embedding 放在时序编码**之后**（不参与时序Transformer）
  - 行业是静态属性，不随时间变化，不需要进时序编码器
- 放在 CrossStockAttention **之前**
  - 让股票间交互时能看到彼此的行业标签
  - 同行业股票互相参考，跨行业股票各自独立

---

### 4.2 训练流程改动

**修改文件**: `code/src/train.py`

1. **feature_cloums_map**：原 197 个特征名前插入 9 个有效新字段（设计19个，实际10个数据不可用）
2. **行业vocab**：从训练数据中自动发现所有行业名 → 构建 name → idx 映射
3. **数据处理**：`create_ranking_dataset_vectorized()` 返回第5个值 `industry_indices`
4. **模型初始化**：`StockTransformer(..., num_industries=len(vocab)+1)`
5. **vocab持久化**：训练完成后保存 `industry_vocab.pkl`，供预测时复用

---

### 4.3 推理流程改动

**修改文件**: `code/src/predict.py`

1. 优先加载 `stock_data_merged.csv`（合并数据），不存在则回退到 `train.csv`
2. 加载训练时保存的 `industry_vocab.pkl`
3. 构建推理序列时提取行业索引
4. 模型前向传播时传入 `industry_idx`

## 五、兼容性说明

### 5.1 渐次降级

所有新数据维度都是**可选的**。模型在以下任意级别都能运行：

| 数据级别 | 特征维度 | 所需文件 |
|----------|----------|----------|
| 完整版 | ~206 维 | stock_data_merged.csv + industry_vocab.pkl |
| 估值版 | ~201 维 | stock_data.csv (增强版) |
| 基础版 | ~197 维 | train.csv (原有数据) |

当 `stock_data_merged.csv` 不存在时，自动回退到 `stock_data.csv` 或 `train.csv`。

### 5.2 回退机制细节

- **train.py**：检测 `data/stock_data_merged.csv` 是否存在，不存在则用 `data/train.csv`
- **predict.py**：同上
- **行业vocab**：如果 `industry_vocab.pkl` 不存在，`num_industries=2`（仅保留 padding 索引），Embedding 层退化为无害的占位层
- **所有新字段**：如果合并数据中某列缺失，按 0 填充

## 七、注意事项

### 7.1 北向资金已不可用

- 2024年8月19日起，沪深交易所停止每日披露北向资金净买卖数据
- `stock_hsgt_hist_em()` 在此之后的日期返回空数据
- 未在本次改造中包含北向资金字段

### 7.2 baostock 的请求频率限制

- `get_stock_data.py` 中每10只股票暂停2秒
- `stock_fundamental.py` 中每次请求暂停0.1秒
- 不要移除这些延迟，否则可能被 ban IP

### 7.3 季频数据的前向填充不会引入未来信息

- merge_all_data.py 确保只使用 `statDate <= tradeDate` 的财报数据
- 例如：4月15日的交易数据，只会用3月31日或更早发布的季报

### 7.4 Windows 环境特殊说明

- `data/run.sh` 是 Linux/Docker 环境用的
- Windows 直接运行 `python xxx.py` 即可
- `multiprocessing` 需要 `spawn` 模式（已在 predict.py 中设置）

## 九、后续可能的改进方向

## 十、实际运行结果与问题排查

> **记录日期**: 2026-04-25  
> **问题**: 引入多维数据后，模型预测加权收益率反而下降

### 10.1 现象

运行完整 pipeline（抓取 → 合并 → 训练 → 预测 → 评分）后：

| 版本 | 加权收益率得分 |
|------|---------------|
| 改造前（纯量价 197 维） | 0.043 |
| 改造后初版（理论 ~218 维） | **0.021** |

得分下降了约 50%。

### 10.2 排查过程

#### 第一步：检查数据文件完整性

```bash
ls data/*.csv
```

发现：
- `stock_fundamental.csv` — **文件不存在**（`stock_fundamental.py` 从未成功运行）
- `market_data.csv` — 存在，但**列的完整性不足**

#### 第二步：检查合并后数据列

```bash
head -1 data/stock_data_merged.csv
```

合并数据实际列：
```
股票代码, 日期, 开盘, 收盘, 最高, 最低, 成交量, 成交额, 振幅, 涨跌额, 换手率, 涨跌幅,
市盈率TTM, 市净率MRQ, 市销率TTM, 市现率TTM,  ← 4列估值 ✓
行业名称,                                     ← 行业名 ✓
SHIBOR_ON, SHIBOR_1W(未用), SHIBOR_1M(未用), QVIX, 涨停家数, 连板均值, 封单资金均值  ← 仅5列有值
```

而 `feature_cloums_map['158+39']` 还声明了：
```
ROE_avg, 净利率, EPS_TTM, 净利同比, 权益同比, 资产负债率,  ← 6列基本面 全0
CPI_同比, CPI_环比, PMI_制造业, M2_同比                    ← 4列宏观 全0
```

#### 第三步：定位根因

**10 列"死特征"**：代码声明了 10 个特征列，但合并数据中根本没有这些列（或被 fillna(0) 填充为全零）。

| 死特征 | 原因 | 后果 |
|--------|------|------|
| `ROE_avg`, `净利率`, `EPS_TTM`, `净利同比`, `权益同比`, `资产负债率` | `stock_fundamental.py` 从未运行 | 6 列全 0 |
| `CPI_同比`, `CPI_环比`, `PMI_制造业`, `M2_同比` | akshare CPI/PMI/M2 API 远程抓取失败 | 4 列全 0 |

#### 第四步：理解为何全零列会降低模型性能

```
数据流：
1. 合并 CSV 中缺失列 → 代码 fillna(0) → 全0
2. StandardScaler 拟合全0列 → mean=0, std=0 → transform 后 (0-0)/0 = NaN → fillna(0) → 仍是全0
3. 模型 input_dim = len(features) = 217（含10列0）
4. 模型 input_proj 层权重矩阵: [217, d_model]
```

**本质问题**：10 列死特征使 `input_dim` 从 207 膨胀到 217。模型的第一层 Linear 多了 `10 × d_model` 个参数，这些参数需要从训练数据中学习"忽略"死维度——但在有限数据下，模型不仅没学会忽略，还被这些维度分散了注意力。

> **直觉类比**：给一个人 10 个恒为 0 的指标让他做判断，这些指标不提供任何信息，但会消耗他的注意力——他会在这些无意义的维度上"寻找规律"（过拟合噪声），从而忽略真正有价值的维度。

### 10.3 修复方案

#### 修复一：移除死特征

从 `train.py` 和 `predict.py` 的 `feature_cloums_map['158+39']` 中**删除 10 个不存在数据的列**，仅保留真实有数据的 9 个新特征：

```python
# 修复后的 feature_cloums_map['158+39']（仅新增部分）
'instrument', '开盘', '收盘', '最高', '最低', '成交量', '成交额', '振幅', '涨跌额', '换手率', '涨跌幅',
'市盈率TTM', '市净率MRQ', '市销率TTM', '市现率TTM',        # 估值 4列 ✓
'SHIBOR_ON', 'QVIX', '涨停家数', '连板均值', '封单资金均值',  # 市场情绪 5列 ✓
'KMID', 'KLEN', ...  # Alpha 158列 + 技术指标 39列
```

修复后实际特征维度：197 + 4 + 5 = **206 维**（不含行业 Embedding）

#### 修复二：增加容错补全逻辑

在特征工程完成后，对 `feature_cloums_map` 中数据缺失的列自动填 0：

```python
# train.py / predict.py 中新增（位于特征工程之后）
for col in feature_columns:
    if col not in processed.columns:
        processed[col] = 0.0
```

这确保：即使数据源部分失败，pipeline 也不会因 KeyError 崩溃，而是以全零列降级运行。

### 10.4 经验教训

1. **声明特征前先验证数据存在**：`feature_cloums_map` 中的特征必须与实际数据文件的列一致。每新增一个维度都应先抓取成功、检查 CSV 列名、再纳入特征映射。

2. **全零特征不是无害的**：直觉上"全是 0 的特征对模型无影响"是错误的。神经网络在有限数据下会在全零维度上过拟合噪声，降低有效维度的学习效率。

3. **pipeline 应有自检机制**：理想情况下，训练前应自动检测哪些列全为常数（方差为 0），并报警或自动移除。这是后续可加的改进。

4. **akshare 接口不稳定是已知风险**：宏观数据的 CPI/PMI/M2 接口在实际运行中可能返回空数据。代码层面的容错和降级是必要的。

## 十一、第二轮排查：预测-评估日期错位

> **记录日期**: 2026-04-25  
> **问题**: 移除死特征后重训，测试得分反而变为负数（-0.00085）

### 11.1 现象

| 版本 | 训练 final_score | 测试加权收益率 |
|------|-----------------|---------------|
| 改造前（纯量价） | ~0.17? | 0.043 |
| 改造后 + 死特征（218维） | 0.1633 | 0.021 |
| 移除死特征后（206维） | **0.1444** | **-0.00085** |

训练指标正常（0.14），但测试得分竟然是负数，说明模型选出的股票在测试期整体下跌。

### 11.2 排查：追溯数据流

关键发现 —— 两套数据的**日期范围不一致**：

```
stock_data_merged.csv  日期范围: 2024-01-01 ~ 2026-04-20  (merged_csv 拉取时间更长)
train.csv              日期范围: 2024-01-02 ~ 2026-03-06  (训练截止日)
test.csv               日期范围: 2026-03-09 ~ 2026-03-13  (评估窗口，共5个交易日)
```

`predict.py` 原逻辑：
```python
# 优先用合并数据 → latest_date = 2026-04-20
merged_file = 'data/stock_data_merged.csv'
if os.path.exists(merged_file):
    data_file = merged_file  # ← 用了这个
```

`predict.py` 取 `raw_df['日期'].max()` 作为预测基准日，得到 **2026-04-20**。

`score_self.py` 用 `test.csv` 计算收益率，评估区间是 **2026-03-09 ~ 2026-03-13**。

**后果**：模型基于 4 月下旬的市场环境选股，但被评估在 3 月中旬的表现。两者差了 6 周——A 股在这个时间跨度内可能经历了完全不同的市场风格和板块轮动。

### 11.3 根因图解

```
模型看到的"今天"          实际评估的"未来5天"
    │                         │
    ▼                         ▼
━━━━┿━━━━━━━━━━━━━━━━━━━━━━━━┿━━━━
2024-01    2026-03-06   2026-03-13   2026-04-20
              train截止    test窗口     predict基准日
                                         ↑
                                  错位！模型在预测
                                  1.5个月后的行情
```

正确的对应关系应该是：
```
   predict 基准日 = 2026-03-06 (train 最后一天)
   → 预测 2026-03-09 ~ 2026-03-13 的股票排序
   → score_self.py 用 test.csv 评估同一区间
```

### 11.4 为什么第一次（0.0206）不是负数

纯属运气。虽然日期也错位了，但那次训练恰巧选出了几只跨期都涨的股票。第二次训练（不同随机种子）学到了不同的排序偏好，选出在3月表现不好的股票。

这也是为什么得分从 0.0206 掉到 -0.00085 —— 两次训练结果的**方差**叠加在日期错位的**偏差**之上。

### 11.5 修复

[predict.py](code/src/predict.py) 调整数据源优先级：**自评场景优先用 `train.csv`，因其日期截止到训练期最后一天，与评估窗口天然对齐**：

```python
# 修复后
train_file = 'data/train.csv'
merged_file = 'data/stock_data_merged.csv'
if os.path.exists(train_file):
    data_file = train_file           # ← 优先：日期正确，且含全部多维特征
elif os.path.exists(merged_file):
    data_file = merged_file          # ← 回退：无 train.csv 时用合并数据
```

`train.csv` 由 `split_train_test.py` 从 `stock_data_merged.csv` 按日期切分而来，包含所有多维特征列（PE/PB、行业、SHIBOR、QVIX 等），只是日期截止到 2026-03-06。用它可以同时满足：
- ✅ 特征完整（多维数据全部保留）
- ✅ 日期对齐（预测基准日 = 评估窗口前一天）

### 11.6 经验教训

1. **预测基准日必须与评估窗口匹配**：自评 pipeline 中 `predict.py` 的输入数据最后一天，必须是测试期开始前的最后一个交易日。

2. **数据源选择不能只看"数据全不全"**：merged CSV 特征最全，但时间范围过大反而导致错位。`train.csv` 特征同样全，且日期范围正确。

3. **负分不一定是模型差**：-0.00085 说明预测与评估时间错位，而非模型学到了反向信号。设计 pipeline 时，数据时间和评估逻辑的对齐比特征数量更基础。

## 十二、第三轮改进：截面分位数特征

> **记录日期**: 2026-04-25  
> **问题**: 修复日期错位后得分仅 0.025，相比原始 0.043 没有突破

### 12.1 为什么市场级特征对排序帮助有限

回顾第二轮修复后保留的 9 个新特征：

```
个股级（有区分度）：市盈率TTM, 市净率MRQ, 市销率TTM, 市现率TTM, 行业Embedding
市场级（无区分度）：SHIBOR_ON, QVIX, 涨停家数, 连板均值, 封单资金均值
```

5 个市场级特征在同一交易日对所有 300 只股票取值完全一样。模型做的是**横截面排序**——"今天这 300 只里谁未来 5 天涨最多"。对所有股票都一样的值，对排序不提供信息。

4 个估值特征虽有区分度，但存在两个局限：
1. **行业间不可比**：银行 PE≈5，科技 PE≈50+，绝对值跨行业比较没有意义
2. **时序变化极慢**：PE/PB 在 45 天窗口内几乎不变，对短期预测信号弱

### 12.2 改进方案：截面分位数排名

对每个交易日，计算每只股票在当日全部股票中的估值百分位：

```python
def _add_cross_sectional_rank(processed, val_cols):
    for col in ['市盈率TTM', '市净率MRQ', '市销率TTM', '市现率TTM']:
        processed[col + '_分位'] = processed.groupby('日期')[col].rank(pct=True)
```

**为什么百分位比原始值更好**：

| 对比 | 原始 PE | PE 分位数 |
|------|---------|----------|
| 银行股 | 3.7 | 0.12（比88%的股票便宜） |
| 科技股 | 55.0 | 0.78（比22%的股票贵） |
| 跨行业可比 | 否——3.7 vs 55 无法直接比 | 是——0.12 vs 0.78 一目了然 |
| 时序稳定性 | 差——PE 随股价每日波动 | 好——相对排名变化更平滑 |
| 量纲统一 | 否——4 个指标量纲不同 | 是——全部归一化到 [0,1] |

> **直觉**：告诉模型"这只股票的 PE 是 12"不如告诉它"这只股票的 PE 比全市场 85% 的股票都低"。后者是一个天然的、跨行业可比的"便宜/贵"信号。

### 12.3 实现位置

在 [train.py](code/src/train.py) 和 [predict.py](code/src/predict.py) 的特征工程之后、Scaler 之前插入：

```
特征工程（per-stock）→ concat → 补全缺失列 → 截面分位数 → StandardScaler
```

新增 4 列：`市盈率TTM_分位`, `市净率MRQ_分位`, `市销率TTM_分位`, `市现率TTM_分位`

特征总数从 206 升至 **210 维**（含 Embedding 则 211 维）。

### 12.4 为什么这个方向更有希望

- **全是纯个股级特征**：每个交易日的百分位对每只股票取值不同，无市场共性噪声
- **零额外数据依赖**：不需要新的数据源，完全从已有的 PE/PB 列派生
- **天然的归一化**：百分位天然在 [0,1] 区间，不受极端值（PE=1000+）影响
- **信息增量明确**：原始值与分位值的相关性 < 1.0，提供了新的排序信息

## 十三、第四轮改进：扩展数据 + 时间衰减 + 移除市场级死特征

> **记录日期**: 2026-04-25
> **问题**: 截面分位数改进后得分 0.025，仍低于基线 0.043

### 13.1 改动内容

#### 改动一：扩展历史数据范围

| 文件 | 改动 |
|------|------|
| `get_stock_data.py` | `start_date` 从 `"2024-01-01"` 改为 `"2021-01-01"` |
| `data/split_train_test.py` | `--train-start` 默认值从 `"2024-01-02"` 改为 `"2021-01-04"` |
| `data/market_data.py` | 自动跟随 stock_data.csv 日期范围，无需手动改 |

训练数据从 ~400 个交易日扩展到 ~1150 个交易日。

#### 改动二：时间衰减样本权重

旧数据的市场模式可能与当下差异较大，直接等权训练会导致旧噪声干扰模型。

**实现**：在 `utils.py` 的 `create_ranking_dataset_vectorized()` 中，为每个样本计算时间衰减权重：

```
weight = time_decay ^ ((max_date - sample_date).days / 365)
```

- `config.py` 新增 `time_decay: 0.5`（1年半衰期）
- 训练集 `time_decay=0.5`，验证集 `time_decay=1.0`（不衰减）
- 权重在 `train.py` 的训练循环中乘到每个样本的 loss 上

| 日期 | 距今天数 | 权重 |
|------|---------|------|
| 2026-03 | 0 | 1.00 |
| 2025-03 | 1年 | 0.50 |
| 2024-03 | 2年 | 0.25 |
| 2023-03 | 3年 | 0.125 |
| 2022-03 | 4年 | 0.06 |
| 2021-03 | 5年 | 0.03 |

#### 改动三：移除5个市场级同值特征

SHIBOR_ON、QVIX、涨停家数、连板均值、封单资金均值——这5个特征在同一交易日对所有300只股票取值完全相同。模型做的是**横截面排序**，对全体股票相同的值对排序零贡献，只增加噪声维度。

从 `train.py` 和 `predict.py` 的 `feature_cloums_map['158+39']` 中移除。

特征维度：210 → **205**（197基础 + 4估值 + 4截面分位）

#### 改动四：清理 akshare 宏观接口

从 `data/market_data.py` 中移除 `fetch_macro_cpi()`、`fetch_macro_pmi()`、`fetch_macro_m2()` 三个从未成功运行的函数，避免无效报错。

### 13.2 运行结果

| 版本 | 特征维度 | 测试加权收益率 | 训练final_score | best epoch |
|------|---------|--------------|----------------|------------|
| 改造前(纯量价) | 197 | **0.043** | ~0.17 | 6 |
| +死特征 | 218 | 0.021 | 0.1633 | — |
| 移除死特征 | 206 | -0.00085 | 0.1444 | — |
| 修复日期+截面分位 | 210 | 0.025 | — | — |
| 扩展数据+时间衰减 | 210 | 0.0035 | 0.2299 | 39 |
| **移除市场级特征** | **205** | **0.0164** | — | — |

### 13.3 分析

1. **扩展数据有效缓解了过拟合**：best epoch 从 6 推迟到 39，训练 final_score 提升到 0.23，说明更多数据让模型学到了更稳定的模式。

2. **但泛化仍然不足**：训练 0.23 vs 测试 0.016，差距巨大。核心矛盾是模型在训练期学到的排序规律无法迁移到测试期（2026-03-09~13）。

3. **市场级特征移除有正面效果**：从 0.0035 提升到 0.0164（3.7倍），验证了"同值特征对排序无用"的判断，但仍未回到基线 0.043。

4. **仍未超越基线的可能原因**：
   - 基线 0.043 是在 2024-2026 数据上训练的，数据分布与测试期更接近
   - 引入 2021-2023 数据后，虽然加了时间衰减，但旧数据的分布偏移仍可能拉低模型对近期模式的拟合精度
   - 4个估值特征（PE/PB/PS/PCF）和4个截面分位特征可能本身信号弱，甚至引入噪声

### 14.2 新增数据源逐项评判

#### 一、日频估值 PE/PB/PS/PCF — ✅ 有用

**来源**：baostock K线接口内置字段，零额外API开销

**实验证据**：对比 #9（含估值+分位，205维，0.0264）vs #10（纯量价197维，0.007），差距3.8倍。

**为什么有用**：
- 估值是**个股级特征**，每只股票值不同，对横截面排序有区分度
- PE低=便宜，PB<1=破净，这些是A股经典选股因子
- baostock内置字段，数据完整无缺失

**局限**：
- 行业间不可比（银行PE≈5，科技PE≈50），需配合截面分位使用
- 时序变化慢，短期信号偏弱

**迁移建议**：**直接加入队友模型**，零风险。但一定要配合截面分位排名使用（见下文）。

#### 二、行业分类 Embedding — ⚠️ 有用但需谨慎

**来源**：baostock `query_stock_industry()`，申万一级行业（~28类）

**实验证据**：本项目无法做对比实验（代码已深度融合），但理论上行业信息对排序有两个好处：
1. 行业内相对排序（同行业谁更好）
2. 板块轮动信号（当前什么行业强）

**为什么谨慎**：
- 行业是**静态属性**，不随时间变化，时序模型难以利用
- Embedding 层增加了 ~18K 参数，小数据下容易过拟合
- 如果模型架构不支持额外的条件输入（如纯MLP模型），强行加入会破坏原有结构

**迁移建议**：如果队友模型有条件输入机制，可以加。如果是简单模型，建议用**行业哑变量**或**行业排名特征**代替 Embedding。

#### 三、季频基本面 ROE/净利率/EPS/成长性/偿债 — ❌ 不可用

**来源**：baostock 财务报表 API

**实验结果**：`stock_fundamental.py` 从未成功运行，数据文件不存在。

**根因**：baostock 的 `query_profit_data()`、`query_growth_data()`、`query_balance_data()` 接口对沪深300成分股大量返回空数据。

**迁移建议**：**不建议投入时间**。如果想用基本面数据，换数据源（如 tushare Pro 或 Wind），但成本高。

#### 四、宏观指标 CPI/PMI/M2 — ❌ 不可用

**来源**：akshare 国家统计局/央行

**实验结果**：akshare 接口远程抓取反复失败。

**根因**：akshare 的宏观经济接口依赖网页爬虫，国家统计局网站结构经常变化导致接口失效。

**迁移建议**：**不建议使用 akshare 宏观接口**。如果确实需要，考虑：
- 手动下载 CSV 固化到本地
- 用 tushare Pro 的宏观接口（付费但稳定）
- 但即使拿到了数据，CPI/PMI 是月频、M2是月频，对日频排序贡献极低

#### 五、市场情绪 SHIBOR/QVIX/涨停统计 — ❌ 对排序无用

**来源**：akshare，接口稳定可抓取

**实验证据**：对比 #4（含市场级特征，210维，0.025）vs #9（去市场级，205维，0.0264），去掉后反而略好。

**为什么无用（关键教训）**：
- 这5个特征在同一交易日对**全部300只股票取值完全相同**
- 模型的任务是**横截面排序**——"今天300只里谁涨最多"
- 对所有股票相同的值 = 对排序零区分度 = 纯噪声
- 模型要花参数去"忽略"这些维度，分散了有效特征的学习

**迁移建议**：**绝对不要加入横截面排序模型**。如果做的是时序预测（预测大盘涨跌），才有用。

#### 六、截面分位数排名 — ✅✅ 最有价值的新特征

**来源**：从已有估值列派生，零额外数据依赖

**实现**：每个交易日，计算每只股票估值在全市场中的百分位排名

```python
processed['市盈率TTM_分位'] = processed.groupby('日期')['市盈率TTM'].rank(pct=True)
```

**为什么比原始值好**：

| 对比 | 原始 PE | PE 分位数 |
|------|---------|----------|
| 银行 PE=5 vs 科技 PE=55 | 跨行业不可比 | 0.12 vs 0.78，一目了然 |
| 量纲 | 不同指标量纲不同 | 全部归一化到 [0,1] |
| 极端值 | PE=1000+ 拉偏分布 | 百分位天然抗极值 |

**迁移建议**：**必加**。对队友模型而言，这是ROI最高的改动——不需要新数据源，只需一行代码，让估值特征跨行业可比。

### 14.3 七条核心经验

#### 经验一：全零特征不是无害的

声明了数据不存在的特征列，fillna(0)后全零，直觉上"对模型无影响"。实际上：
- 全零列使 input_dim 膨胀，模型第一层多了无用参数
- StandardScaler 对全零列：mean=0, std=0 → transform 后 NaN → fillna(0) → 还是全零，但Scaler的参数空间被污染
- 有限数据下，模型在全零维度上过拟合噪声

**规则**：feature_columns 中只放数据真实存在的列。声明前先 `df[col].std() > 0` 检验。

#### 经验二：同值特征对排序任务无用

横截面排序任务中，同一交易日所有股票取值相同的特征（SHIBOR、QVIX、涨停数等）对"选谁"不提供信息，只增加维度和噪声。

**规则**：排序模型的特征必须是**个股级**的（每只股票值不同），而非市场级或行业级的共用值。

#### 经验三：预测基准日必须与评估窗口对齐

predict.py 取 `raw_df['日期'].max()` 作为预测基准日。如果用 merged CSV（截止到4月），模型基于4月的市场环境选股，但评估在3月，选出的股票在3月表现可能很差。

**规则**：自评场景下，predict 的输入数据最后一天 = 测试期开始前的最后一个交易日。

#### 经验四：扩展旧数据不一定有帮助

3年旧数据（2021-2023）+ 时间衰减 = 测试得分下降（0.0264 → 0.0164）。原因：
- 旧市场风格与当下差异太大（疫后复苏、地产暴雷 vs 2026市场）
- Scaler 在全量数据上拟合，被旧数据分布拉偏
- time_decay 只缩小旧样本的梯度，但不消除其方向上的冲突

**规则**：对于短期预测任务，近期数据的分布对齐比数据量更重要。扩展数据前先确认旧数据与当前市场的分布是否相似。

#### 经验五：正则化不是越大越好

dropout 0.3 + weight_decay 1e-3 让得分从 0.0164 降到 0.0102。模型容量（d_model=128, 2层）本来就不大，过强的正则化让它学不到排序信号。

**规则**：正则化强度应与模型容量和数据量匹配。小模型+中等数据，dropout 0.1~0.2 足够。

#### 经验六：评估指标方差极大

5个交易日 × 5只股票 × 等权0.2 = 最终得分。一只股票涨跌5%就导致总分波动0.01。0.043 和 0.007 的差距可能只是选了不同的股票。

**规则**：单次评估得分不可靠，应多次训练取平均，或关注排名一致性而非绝对分数。

#### 经验七：等权0.2浪费排序信息

模型输出的分数差距（如第1名 0.8 vs 第5名 0.3）包含了置信度信息，但等权0.2把第1名和第5名等同对待。softmax 按分数分配权重可以让高置信度的选股获得更大影响。

**规则**：如果赛制允许非等权，用 softmax 分配权重。
